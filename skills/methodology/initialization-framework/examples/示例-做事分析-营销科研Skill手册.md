# 做事分析 · 营销科研 Skill 手册

> 输入项目：做一本 AI 驱动的"市场营销科研 Skill 手册"，让团队任何人都能用一个 Skill 把一个营销问题从模糊问到可执行方案一次走完。

---

## 一、开钻 · 天花板

**为什么是这本手册？**

如果不做，会失去什么？

- 团队每接一个营销问题，都要从零搭框架——同一件事问三遍得出三套答案，对外口径分裂，对内精力分散。
- 没有统一工具，每个 Agent 各自为战，最后交付物的质量取决于当天谁有空、谁心情好。
- 营销科研的核心资产不是数据，是判断。判断不沉淀，团队永远在重复造轮子。

做成了，会带来什么？

- 团队对"什么是营销科研"形成单一共识——一套话、一个流程、一组验证标准。
- 把分散在十几个人脑子里的判断结构化进 Skill，让新人第一天就能输出老手水准。
- 形成可复用的科研流水线，下一次接单只需要换输入，不换骨架。

**天花板确认：** 这本手册不是文档，是团队的"判断操作系统"。值不值得做，看的是这个操作系统能不能把"判断"从个人资产升级为团队资产。

裂缝：手册能定义术语，但术语之间怎么咬合？谁来保证每层的对齐？这就是第二层要解决的。

## 二、话语体系 · 翻译层

营销科研有四套话，每套面对一个对象：

### 对领导：价值语言

> "我们用一套穿透式分析框架，把营销问题的判断从个人能力变成团队基础设施。下一次再遇到同类问题，复用率 80% 以上，决策时间压缩一半。"

关键词：**复用率、决策时间、团队资产**。领导不关心方法论，关心 ROI 和可复制性。

### 对团队：方法语言

> "接到任何营销问题，先跑六层穿透——价值判断 → 话语体系 → 实现路径 → 验证标准 → 调度编排 → 扩展演化。每层只回答'这一层地面是什么'，不并列、不铺陈。"

关键词：**六层穿透、单刀直入、层层穿透**。团队需要的是可执行的方法论。

### 对用户：结果语言

> "你给我一个营销问题，我给你一份穿透式分析报告——告诉你这事值不值得做、怎么做、怎么算做成、谁来做、用完能不能复用到下一个问题。"

关键词：**穿透式分析、值不值得、能不能复用**。用户不关心层数，关心"你能不能想透"。

### 对学术界：理论语言

> "基于消费者行为、说服心理、品牌资产的整合视角，构建一个六层穿透式分析框架，强调判断的可复用性而非单次输出。"

关键词：**消费者行为、说服心理、品牌资产、可复用性**。学术圈认的是理论锚点。

**统一术语三个：**

1. **穿透式分析**——从价值判断到扩展演化的纵向钻探，不是横向并列。
2. **价值判断**——问"不做会失去什么"，不是"做成了会得到什么"。
3. **验证标准**——回答"怎么判断做成了"，不是"做到哪一步了"。

裂缝：术语统一了，但手册怎么组织？是按六层编排，还是按问题类型编排？这就是第三层。

## 三、实现路径 · 施工图

**方法选择：** 三段式交付——元方法手册（这本）+ 场景化子手册（每个营销场景一个）+ 活水反馈机制（季度回炉）。

**为什么选三段式？**

- 单层手册太抽象，团队用不起来。
- 纯场景化手册没有共同骨架，换场景就要重学。
- 没有反馈机制，手册会过时。三段式解决三个问题。

**推进节奏：**

| 阶段 | 时间 | 重点 |
|---|---|---|
| 骨架期 | 第 1-2 周 | 把六层框架写成可执行规范，每层给一个示范案例 |
| 场景期 | 第 3-6 周 | 选 3 个高频场景（新品上市、品牌焕新、转化漏斗）做子手册 |
| 活水期 | 第 7 周起 | 每月一次回炉，收集使用反馈，每季度大改一次 |

**资源配置：**

- 1 个主笔（懂方法论 + 能写）
- 2 个场景负责人（每个场景 1 人）
- 1 个反馈官（专职收集使用反馈）
- 1 个工具官（维护 Skill 工程化）

**关键节点：**

- 第 2 周末：六层骨架必须有 1 个完整示范案例，否则下游场景开发无法启动。
- 第 6 周末：3 个场景子手册必须可独立使用，否则"复用"承诺落空。
- 第 8 周末：第一次回炉必须给出"哪些层失效、为什么"的诊断，否则活水期变成摆设。

裂缝：施工图有了，但怎么判断它做成了？手册写完了不等于手册有用了——团队用不用得起来才是验证标准。

## 四、验证标准 · 验收单

### 成功指标 · 量化

**底线指标（达不到算失败）：**

1. 团队任意成员拿一个新营销问题，按手册走完六层，输出报告 ≤ 4 小时。
2. 同一问题不同人跑手册，输出的"价值判断"层重合度 ≥ 70%。
3. 手册 v1 发布后 30 天内，团队实际使用率 ≥ 60%。

**优秀指标（达到算超额）：**

1. 使用手册产出的方案在客户/老板那里的"穿透感评价"≥ 4 分（5 分制）。
2. 手册被复用到非营销场景（产品决策、个人决策）≥ 3 次。
3. 团队能主动提出手册哪一层失效，并给出修正方案——手册开始自我演化。

### 验证流程

- **每章节写完**：主笔 + 1 个外部读者盲测，独立走一遍，对比输出。
- **整本发布前**：3 个场景各跑 1 个真实问题，看手册能不能扛住实战。
- **发布后每月**：反馈官收集 10 条使用反馈，归类到六层中"最常失效的层"。

### 风险预案

| 风险 | 概率 | 应对 |
|---|---|---|
| 团队嫌麻烦不用 | 中 | 把手册嵌入既有流程，不额外增加步骤 |
| 场景覆盖不全 | 高 | 首批只做 3 个高频场景，其他场景 v2 再补 |
| 手册写得太抽象 | 中 | 每层必须有 1 个真实案例，不能只有定义 |
| 反馈机制流于形式 | 高 | 把"提出失效层"列入 OKR，反馈官有否决权 |

裂缝：验收单有了，但谁来跑这套验收？主编？反馈官？还是所有 Agent 协同？这就是第五层。

## 五、调度编排 · 多Agent协同

**调度机制：**

- **主编 Agent**：负责六层穿透的最终质量把关，对每层报告有一票否决权。
- **场景 Agent**：每个场景 1 个 Agent，负责把六层翻译成该场景的具体操作。
- **反馈 Agent**：每月跑一次"哪层失效最多"的诊断，输出诊断报告。
- **工具 Agent**：维护 Skill 工程化，确保手册可被实际调用，不是 PPT 文档。

**任务依赖：**

```
主编 (价值判断+话语体系)
   ↓
场景 Agent (实现路径+验证标准)
   ↓
反馈 Agent (调度优化+扩展演化)
   ↓
工具 Agent (封装+版本管理)
```

**权重规则 · 冲突决策：**

1. **质量冲突**（主编 vs 场景 Agent）→ 主编赢。
2. **场景适配冲突**（场景 Agent vs 反馈 Agent）→ 反馈 Agent 的数据赢。
3. **工具冲突**（工具 Agent vs 其他）→ 看影响范围，影响 > 3 个场景的，工具 Agent 赢；否则其他 Agent 赢。

**职责边界：**

| Agent | 决策 | 执行 | 不做 |
|---|---|---|---|
| 主编 | 框架层定义、术语统一 | 写骨架、做示范 | 不写场景化操作 |
| 场景 Agent | 场景内优先级 | 写子手册、跑实战 | 不动六层骨架 |
| 反馈 Agent | 失效层诊断 | 收集反馈、跑回炉 | 不写手册内容 |
| 工具 Agent | 封装形式、版本号 | 写代码、做部署 | 不评价内容质量 |

**判断标准：** 每个 Agent 知道自己的"不做"。边界不清的协作是分工表，不是调度编排。

裂缝：当下跑通了，但这套方法怎么复用到下一个项目？手册本身会不会过时？

## 六、扩展演化 · 长期视角

**迁移能力：**

1. 这套"六层穿透 + 反馈回炉"可以复用到：
   - 产品决策（新功能做不做）
   - 内容生产（一篇深度文章怎么写）
   - 个人决策（要不要换工作）
   - 团队管理（一次组织调整怎么做）
2. 复用时调整的核心是"输入"和"验证标准"——骨架不变。

**迭代机制：**

- **月度小回炉**：收集 10 条反馈，归类到六层，更新"失效层清单"。
- **季度大改版**：根据失效层清单，重写最失效的 1-2 层。
- **年度重构**：六层结构本身是否需要调整？比如要不要加第七层"风险层"？

**淘汰规则：**

- 触发条件 1：连续 2 个季度，团队使用率 < 30% → 重构手册，而非放弃。
- 触发条件 2：连续 3 个季度，使用率 < 30% → 放弃这套手册，换方法论。
- 触发条件 3：六层中某一层连续 4 个月被标为"最失效" → 删掉该层或合并到其他层。

**自我演化机制 · Dogfood：**

这套手册用来设计它自己的演化。具体来说：

- 每月的反馈回炉，本质是"用扩展演化层的方法优化手册本身"。
- 每季度的诊断报告，本质是"用验证标准层的方法评估手册本身"。
- Dogfood 失败 = 手册失效。

## 七、终点 · 沉默片刻

这本手册的天花板是"把判断从个人资产升级为团队资产"。

但有个真相必须说出来：

**手册再好，团队不用，就是一张废纸。**

所有六层穿透、所有术语统一、所有验收标准、所有调度编排，最后都要回答一个问题：

**团队愿意每天用吗？**

如果答案是"不愿意"——那这本手册从价值判断那层就立不住。

这不是方法论的问题，是人的问题。

而人，不是手册能解决的。
